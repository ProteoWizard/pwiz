Osprey DIA pipeline workflow

Top-to-bottom data flow from mzML + spectral library to BiblioSpecLite (.blib) output, drawn around the join / per-file-fan-out boundaries that matter for HPC distribution. The four --task workers — PerFileScoring → FirstPassFDR → PerFileRescoring → SecondPassFDR — can each run as a separate HPC node; the banner heading each task names the files it reads and writes, including the sidecars the next task rehydrates from. The C# implementation (Osprey) is end-to-end cross-impl bit-identical to the Rust reference (osprey) from Stage 1 through Stage 7 + .blib on the Stellar (--resolution unit) and Astral (--resolution hram) 3-file datasets; box color shows per-stage port status. Per-stage parity detail and end-to-end performance are in the tables near the bottom of the page.

Cross-impl parity (Stellar --resolution unit, Astral --resolution hram): Stages 1-4 PIN features bit-identical at ULP on Stellar (466K entries, all 21 features under 1e-10); on Astral 19 of 21 are ULP and xcorr / sg_weighted_xcorr drift at ~1e-7 from the intrinsic f32 HRAM preprocessed-bin cache (matches upstream osprey-scoring). Stages 5 and 6 pass byte-equality at every dump on both datasets (3-file). Stage 7 protein FDR matches at 1e-9 absolute (Stellar 6204/6204, Astral 11779/11779 protein groups); Stage 7 .blib output matches at the SQL row+column level (Stellar 45153 + Astral 129352 RefSpectra, byte-identical RefSpectraPeaks blobs).

End-to-end performance (3-file regression, Windows native, C: SSD source data, AV-scan excluded; 8 cells, median-of-3): Stage 5/6 boundaries are aligned cross-impl so the per-stage cells compare like-for-like: Rust labels stage5 = percolator + protein FDR and stage6 = reconciliation + rescore + gap-fill, while Osprey logs the reconciliation portion under its [TIMING] markers, which Measure-Pipeline.ps1 shifts into stage6. Run-to-run variance is tight on C# (Astral total ~17:44–17:48, Stellar ~3:32–3:46) and wider on Rust, mostly in blib write. These figures are the default sequential per-file run; under --parallel-files the per-file scoring and rescore fan out across files and the wall drops further.

Per-stage commentary:

  • stage1to4 (per-file scoring): C# faster on both datasets (0.68× Stellar, 0.84× Astral). The hot path uses an LOH-allocation-free ~100K-bin XcorrScratchPool, a pre-preprocessed XCorr cache with f32 narrowing, a pooled visitedBins for O(n_fragments) clear, per-file thread scaling, and an mzML read gate that serializes disk I/O across concurrent ProcessFile calls.
  • stage5 (first-pass FDR — percolator + protein FDR; reconciliation planning is folded into stage6 per the alignment above): C# 14% faster on Stellar, 28% on Astral. The SVM dual-coordinate-descent inner loops are SIMD-vectorized (System.Numerics.Vector<double>), the KDE binning in PepEstimator.Fit is parallelized, and the grid search + folds run under OspreyParallel.For with a per-worker SvmTrainScratch pool. At Astral scale the wall is dominated by the percolator FDR loop itself.
  • stage6 (per-file rescore + gap-fill + reconciled parquet write-back, aligned to include reconciliation planning): tied on Stellar (0.99×), C# 33% faster on Astral. The LOESS planner's per-x-point local regressions run under Parallel.For and the rescore reuses the pooled-scratch XCorr path; in default sequential mode reconciliation planning dominates the Astral wall. Under --parallel-files the per-file rescore (--task PerFileRescoring) fans out across files as well, byte-identically.
  • stage7 (second-pass Percolator + protein FDR): C# faster than Rust everywhere. Heavily SVM-dominated — the 2nd-pass FDR re-runs Percolator on the reconciled feature pool with the same SIMD LinearSvmClassifier.Train + parallel KDE as stage5. Protein parsimony's subset-elimination uses HashSet<string>; the 2nd-pass FDR sidecar overlay (FdrScoresSidecar.TryReadOverlay) reads only the sidecar and overlays scores by entry_id without re-reading the source parquet. (Osprey emits the 2nd-pass Percolator work under a separate [STAGE-WALL] second-pass-fdr marker; the totals above combine it with stage7 so the column compares like-for-like with Rust's single stage7 marker.)
  • blib write: C# faster on both datasets. Per-spectrum Ionic.Zlib compression runs under Parallel.For (BlibWriter pre-compresses mzs + intensities); the SQLite insert stays sequential and the output is byte-identical to Osprey's blib.

WSL Linux end-to-end performance — ext4 inside /home (median-of-3): Same workstation as the Windows benchmark above, WSL2 Ubuntu 22.04, .NET 8.0.421, Rust 1.95.0. The WSL ext4.vhdx is on the C: SSD; test data is staged inside the VHDX at /home/brendanx/test/osprey-runs, so every read and write goes through Linux ext4 with no 9P drvfs crossing. This is the “Linux-native” configuration. C# is ahead by 7% on Stellar (3:41 vs 3:58) and 4% on Astral (15:57 vs 16:38) at the total wall. The gap is tighter than the Windows-native or /mnt/c cells because of an asymmetric benefit at stage6 on native Linux ext4: Rust stage6 drops from 5:57 (/mnt/c) to 4:05 (/home) –32%; C# stage6 drops from 5:36 to 4:39 –17%. C#'s stage6 is largely CPU-bound; Rust's remaining stage6 work is more I/O-heavy and benefits more from the native filesystem. Stage 5 (aligned) is tied on both datasets (0.97× Stellar / 0.91× Astral); stage7 is decisively C#-faster (0.93× / 0.78×). Cross-OS output parity is gated separately by Test-Snapshot.ps1 at 1e-6 tolerance on four median-polish columns (median_polish_cosine, _residual_ratio, _min_fragment_r2, _residual_correlation) because both runtimes delegate Math.Log and Math.Exp to the host libm (Linux glibc vs Windows ucrt), which differ at f64 ULP level. All other Stage 1-4 columns are bit-equal cross-OS, and the cross-OS divergence in stage5 percolator TSV output is bounded to a handful of formatted-float characters per ~200 MB file.

Methodology notes. (1) Storage layout. All numbers in the three tables above are on C: SSD physical media; both Windows C:\test\osprey-runs and the WSL ext4.vhdx (now backing both /home and /mnt/c via different paths) live on the same SK hynix 2 TB NVMe. Defender real-time scan exclusions cover both C:\test and the Windows mount points the WSL distro touches; without those exclusions, every disk-bound stage paid a 5-10% scan overhead. (2) Rust dump-writer patch (still load-bearing). Rust's stage-5/6 diagnostic-dump writers in osprey v26.6.0 use unbuffered std::fs::File::create: each writeln! crosses to the kernel. On WSL drvfs/9P that took ~14 min for the 74 MB stage5 percolator dump alone (vs ~1 min for the percolator algorithm itself); BufWriter eliminates this on either OS. The numbers above reflect that patch (committed via PR #36) plus an explicit f.flush(); drop(f); before exit_if_only. (3) C# Astral file parallelism. C# processes files sequentially by default; the WSL Astral run above is therefore the default configuration and matches Rust, which also processes files sequentially. Concurrency is opt-in via --parallel-files [N] (the legacy OSPREY_MAX_PARALLEL_FILES env var remains only as a back-compat cap). Concurrent stage 1-4 over three files peaks at ~60 GB on Astral, which exceeds the headroom left by the Hyper-V + ext4.vhdx machinery on this 64 GB workstation, so the default sequential path is what fits under WSL.

BLIB writer: Osprey's BlibWriter uses DotNetZip / Ionic.Zlib level 6 (the same library Skyline's Util/Extensions/UtilDB.Compress has used since 2009). The Osprey side switched its flate2 backend from pure-Rust miniz_oxide to vendored stock zlib so both sides emit identical Ionic.Zlib-compatible deflate output. Future work: pull Skyline's Model/Lib/BlibData out to Shared/BiblioSpec so Osprey and Skyline share a single C# BLIB writer.
